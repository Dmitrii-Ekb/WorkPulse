# План занятий WorkPulse

Обновлено: **2026-07-20**

## Формат каждой сессии

Работа строится короткими итерациями:

1. Выбирается одна небольшая продуктовая или техническая задача.
2. Ученик сначала формулирует ожидаемое поведение или предлагает решение.
3. Вносится минимальное изменение или выполняется один небольшой эксперимент.
4. Результат проверяется подходящим способом.
5. Ученик объясняет наблюдение своими словами.
6. Наставник уточняет модель происходящего и вводит только необходимую теорию.
7. После законченного этапа предлагается название коммита.
8. Commit и push ученик выполняет самостоятельно.
9. После заметного этапа синхронизируются документы в `docs/learning`.

Проверка выбирается по масштабу изменения:

- после очевидного безопасного однострочного изменения отдельная сборка не обязательна;
- после нескольких связанных изменений выполняется сборка, запуск приложения или тест;
- миграция сначала читается, затем применяется;
- изменение API проверяется реальным HTTP-запросом;
- изменение данных проверяется чтением из API или базы.

Не давать большой готовый блок кода, когда ученик может собрать изменение самостоятельно.

Не считать тему изученной только потому, что конструкция встретилась в коде.

Не повторять стандартные Git-команды без необходимости. Подробно разбирать Git только при ошибке, новом состоянии или новой теме.

## Завершённый этап: асинхронная работа с EF Core

CRUD переводился по одному маршруту:

1. `GET /tasks` — `ToListAsync`;
2. `GET /tasks/{id}` — `FindAsync`;
3. `POST /tasks` — `SaveChangesAsync`;
4. `PATCH /tasks/{id}/complete` — `FindAsync` и `SaveChangesAsync`;
5. `DELETE /tasks/{id}` — `FindAsync` и `SaveChangesAsync`;
6. `GET /tasks/first` — `FirstOrDefaultAsync`.

Разобрано:

- асинхронный обработчик Minimal API;
- `Task`, `async`, `await`;
- освобождение потока во время незавершённого I/O;
- продолжение на любом свободном потоке;
- отличие асинхронности от создания нового потока;
- польза асинхронности для Web API;
- последовательность зависимых операций;
- отсутствие необходимости использовать `await` для `Add`, `Remove` и изменения свойства.

Повторную большую лекцию не начинать без новой практической причины.

## Завершённый этап: ручная практика SQL

В Rider Query Console выполнены:

```sql
SELECT Id, Title, IsCompleted
FROM Tasks;
```

```sql
SELECT Id, Title, IsCompleted
FROM Tasks
WHERE IsCompleted = 0;
```

```sql
SELECT Id, Title, IsCompleted
FROM Tasks
WHERE IsCompleted = 0
ORDER BY Id DESC;
```

```sql
INSERT INTO Tasks (Title, IsCompleted)
VALUES ('SQL test task', 0);
```

```sql
UPDATE Tasks
SET IsCompleted = 1
WHERE Id = 1006;
```

```sql
DELETE FROM Tasks
WHERE Id = 1006;
```

Проверено:

- `SELECT` может возвращать столбцы, константы, вычисления и псевдонимы;
- `WHERE` фильтрует строки;
- `ORDER BY ... DESC` сортирует по убыванию;
- `INSERT` создаёт строку;
- `UPDATE ... WHERE` меняет выбранные строки;
- `DELETE ... WHERE` удаляет выбранные строки;
- `AUTOINCREMENT` не обязан заполнять пропуски;
- Rider показывает количество затронутых строк;
- проверочный `SELECT` после удаления вернул ноль строк;
- тестовая запись удалена;
- production-код не менялся.

Критическое правило:

```text
Не выполнять UPDATE или DELETE без WHERE,
пока изменение всех строк не является осознанной целью.
```

## Завершённый этап: вторая миграция

### Продуктовая задача

Пользователь должен иметь возможность:

- создать задачу только с заголовком;
- необязательно указать описание сразу;
- получить описание при последующем чтении задачи.

### Изменение модели

В `WorkTask` добавлено:

```csharp
public string? Description { get; set; }
```

Разобрано:

- `string?` допускает `null`;
- `null` означает, что описание не задано;
- `= "";` не требуется;
- обязательный `Title` и необязательный `Description` имеют разные контракты.

### Создание и применение миграции

Создана миграция:

```text
AddTaskDescription
```

Перед применением прочитаны методы `Up` и `Down`.

`Up` добавляет:

```text
Tasks.Description → TEXT NULL
```

`Down` удаляет колонку вместе со всеми её значениями.

Выполнено:

```bash
dotnet ef database update
```

Проверено:

- колонка `Description` появилась;
- существующие задачи сохранились;
- существующие задачи получили `Description = NULL`.

### Механика миграций

Разобрано:

```text
текущая C#-модель
        ↕
ModelSnapshot
        ↓
содержимое новой миграции
```

И:

```text
файл миграции
        ↓
database update
        ↓
Up
        ↓
__EFMigrationsHistory
```

Ученик понимает:

- имя миграции само по себе не определяет изменение;
- без изменения модели может появиться пустая миграция;
- EF Core хранит состояние модели в snapshot;
- база хранит список применённых миграций;
- ручное изменение схемы может привести к рассинхронизации.

## Текущий этап: границы HTTP-контракта

### Шаг 9. Входной, выходной DTO и сущность

Текущий `POST /tasks` возвращает `WorkTask` напрямую:

```csharp
return Results.Created($"/tasks/{newTask.Id}", newTask);
```

Цель — понять, является ли сущность подходящим публичным контрактом API.

Порядок:

1. Повторить роли входного DTO, сущности и выходного DTO.
2. Рассмотреть риск: новое свойство сущности автоматически появляется в JSON.
3. Сформулировать требования к ответу задачи.
4. Решить, нужен ли `WorkTaskResponseDto` уже сейчас.
5. При создании DTO выполнить явное преобразование без AutoMapper.
6. Проверить ответы GET и POST.
7. Объяснить назначение отдельного выходного контракта.

Ограничения:

- не добавлять AutoMapper ради нескольких свойств;
- не создавать отдельные слои без практической причины;
- не менять все endpoint-ы одним большим блоком;
- двигаться по одному маршруту.
