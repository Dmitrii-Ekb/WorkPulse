# План занятий WorkPulse

## Формат каждой сессии

Работа строится короткими итерациями:

1. Выбирается одна небольшая продуктовая или техническая задача.
2. Вносится минимальное изменение в проект.
3. Приложение запускается.
4. Проверяется успешный сценарий и хотя бы один ошибочный.
5. Ученик объясняет наблюдение своими словами.
6. Наставник уточняет модель происходящего и вводит только нужную теорию.
7. Выполняется сборка или другая подходящая техническая проверка.
8. Результат фиксируется коммитом после законченного шага.
9. После заметного учебного этапа обновляются документы в `docs/learning`.

Не давать сразу большой готовый блок кода, когда ученик уже знает необходимые части и может собрать решение самостоятельно.

Не считать конструкцию изученной только потому, что она встретилась в готовом коде.

После законченного этапа наставник предлагает название коммита, а ученик самостоятельно выполняет `commit` и `push`.

Отдельно останавливаться на Git-командах нужно только при новой теме Git, непонятном состоянии файлов или ошибке.

## Завершённый этап: конфигурация, организация модели и DTO

### Шаг 1. Строка подключения перенесена в конфигурацию

Выполнено:

- в `appsettings.json` добавлена секция `ConnectionStrings`;
- строка подключения получила имя `WorkPulseDatabase`;
- значение читается через `builder.Configuration.GetConnectionString(...)`;
- значение передаётся в `UseSqlite`;
- добавлена ранняя проверка отсутствующей или пустой настройки;
- проверено, что приложение продолжает работать с существующей базой.

Изучено на практике:

- зачем отделять настройки от кода;
- ключи и секции JSON;
- назначение секции `ConnectionStrings`;
- роль `builder.Configuration`;
- что возвращает `GetConnectionString`;
- почему обязательную настройку полезно проверять при запуске.

### Шаг 2. Проведён эксперимент с ошибкой конфигурации

Выполнено:

- временно изменено имя строки подключения;
- приложение запущено с неправильной настройкой;
- прочитано сообщение `InvalidOperationException`;
- возвращено правильное значение;
- объяснена связь между ключом конфигурации и `UseSqlite`.

### Шаг 3. Модель вынесена из `Program.cs`

Выполнено:

- создан отдельный файл `WorkTask.cs`;
- сохранено пространство имён `WorkPulse`;
- `Program.cs` стал короче;
- CRUD продолжил работать;
- подтверждено, что перемещение класса между файлами не меняет схему базы;
- новая миграция не создавалась.

Цель шага достигнута: начато улучшение организации кода без преждевременного введения слоёв и сложной архитектуры.

### Шаг 4. Входной контракт создания отделён от сущности базы

Выполнено:

- создан `CreateWorkTaskDto`;
- `POST /tasks` принимает только разрешённое поле `Title`;
- сервер самостоятельно создаёт `WorkTask`;
- `IsCompleted` устанавливается сервером в `false`;
- `Id` назначается SQLite;
- проверено, что клиентские `id` и `isCompleted` не управляют сущностью;
- разобрано несовпадение типов `CreateWorkTaskDto` и `WorkTask`;
- выполнено явное преобразование DTO в сущность;
- подтверждено, что DTO не требует миграции;
- проект успешно собран.

Изучено на практике:

- DTO как обычный C#-класс для передачи данных;
- входной контракт HTTP-запроса;
- сущность базы данных;
- разделение API-контракта и модели хранения;
- ограничение набора полей запроса;
- базовая проблема overposting;
- явное преобразование между разными типами.

## Завершённый этап: асинхронная работа с EF Core

### Шаг 5. CRUD переведён на асинхронные операции

Маршруты переносились по одному:

1. `GET /tasks` — `ToListAsync`;
2. `GET /tasks/{id}` — `FindAsync`;
3. `POST /tasks` — `SaveChangesAsync`;
4. `PATCH /tasks/{id}/complete` — `FindAsync` и `SaveChangesAsync`;
5. `DELETE /tasks/{id}` — `FindAsync` и `SaveChangesAsync`;
6. `GET /tasks/first` — `FirstOrDefaultAsync`.

На каждом шаге:

- изменялся только один маршрут;
- добавлялись `async` и `await`;
- запускалось приложение;
- проверялся успешный сценарий;
- проверялся ошибочный сценарий, где он применим;
- подтверждалось, что HTTP-контракт не изменился.

Разобрано на практическом уровне:

- асинхронный обработчик Minimal API;
- `Task` как результат асинхронной операции;
- `await` как ожидание готового результата без удержания потока на всё время ожидания;
- отсутствие необходимости использовать `await` для `Add`, `Remove` и обычного изменения свойства;
- обращение к базе при `ToListAsync`, `FindAsync`, `FirstOrDefaultAsync` и `SaveChangesAsync`;
- последовательное выполнение зависимых асинхронных операций;
- отсутствие гарантии продолжения на том же потоке после `await`.

Этап завершён на уровне работающего кода. Глубокое понимание потоков, пула потоков и механизма продолжения ещё закрепляется вопросами.

## Текущая контрольная точка: вопросы по асинхронности

Перед началом новой темы не добавлять новый код.

Разобрать вопросы ученика о:

- потоке выполнения;
- пуле потоков .NET;
- обработке HTTP-запроса;
- объекте `Task`;
- завершении операции ввода-вывода;
- поведении `await`;
- сохранении точки продолжения;
- продолжении на любом свободном потоке;
- отличии асинхронности от параллелизма;
- пользе асинхронности для Web API под нагрузкой.

Тема считается закреплённой не после переписывания методов, а после того, как ученик сможет своими словами объяснить общую цепочку:

```text
HTTP-запрос
    ↓
поток начинает обработчик
    ↓
запускается операция ввода-вывода
    ↓
await приостанавливает продолжение обработчика
    ↓
поток возвращается в пул
    ↓
операция завершается
    ↓
продолжение получает любой свободный поток
    ↓
формируется HTTP-ответ
```

После вопросов выполнить сборку и зафиксировать законченный этап.

## Ближайшие следующие шаги

### Шаг 6. Увидеть SQL за LINQ

Начать только после того, как ученик закончит вопросы по асинхронности и подтвердит готовность перейти дальше.

- включить или прочитать существующие логи EF Core;
- сопоставить `ToListAsync` с `SELECT`;
- `Add` и `SaveChangesAsync` — с `INSERT`;
- изменение и `SaveChangesAsync` — с `UPDATE`;
- `Remove` и `SaveChangesAsync` — с `DELETE`;
- написать несколько простых SQL-команд вручную отдельно от production-кода.

### Шаг 7. Вторая миграция на реальном изменении модели

Выбрать одно небольшое полезное поле, например:

- описание задачи;
- дата создания;
- срок выполнения;
- приоритет.

Затем:

1. изменить модель;
2. решить, должно ли поле входить в DTO создания;
3. создать новую миграцию;
4. прочитать `Up` и `Down`;
5. применить миграцию;
6. проверить старые и новые данные;
7. объяснить переход схемы из первой версии во вторую.

### Шаг 8. Разделить входные и выходные контракты

После появления практической причины:

- обсудить, нужно ли возвращать сущность `WorkTask` напрямую;
- создать DTO ответа;
- явно сформировать HTTP-ответ;
- понять, какие данные являются внутренними, а какие относятся к публичному API;
- не добавлять AutoMapper до появления реальной необходимости.

### Шаг 9. Добавить автоматическую проверку API

Начать с одного простого интеграционного теста:

- создать тестовый проект;
- запустить приложение в тестовом окружении;
- отправить `POST /tasks`;
- проверить `201 Created`;
- проверить тело ответа;
- проверить ошибочный запрос;
- постепенно покрыть остальные маршруты.

## Правила темпа

- одна новая крупная идея за раз;
- не повторять уже понятую тему без причины;
- при пробеле в базе C# останавливаться и разбирать его на текущем коде;
- новые конструкторы, обобщения, наследование и DI считать только введёнными, пока ученик не сможет объяснить и изменить их;
- не завершать занятие самостоятельно — ученик сам сообщает, когда остановиться;
- сложные термины давать только вместе с практическим примером;
- после ошибки сначала исследовать её, а не сразу скрывать готовым исправлением;
- не вводить репозитории, сервисные слои или контроллеры без конкретной проблемы, которую они решают;
- после заметного этапа синхронизировать `PROGRESS.md`, `PLAN.md` и `ROADMAP.md`;
- не задерживаться на стандартных `commit` и `push`, если Git работает ожидаемо.
