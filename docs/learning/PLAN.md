# План занятий WorkPulse

## Формат каждой сессии

Работа строится короткими итерациями:

1. Выбирается одна небольшая продуктовая задача.
2. Вносится минимальное изменение в проект.
3. Приложение запускается.
4. Проверяется успешный сценарий и хотя бы один ошибочный.
5. Ученик объясняет наблюдение своими словами.
6. Наставник уточняет модель происходящего и вводит только нужную теорию.
7. Результат фиксируется коммитом после законченного шага.

Не давать сразу большой готовый блок кода, когда ученик уже знает необходимые части и может собрать решение самостоятельно.

## Ближайший план: база данных

### Шаг 1. Короткая теория перед практикой

Разобрать без перегруза:

- база данных и СУБД;
- реляционная модель;
- таблица, строка и столбец;
- первичный ключ;
- SQL;
- чем SQLite отличается от PostgreSQL и SQL Server;
- почему SQLite подходит для первого этапа WorkPulse;
- чем EF Core отличается от самой базы данных.

### Шаг 2. Подключить SQLite и EF Core

- установить пакеты;
- проверить изменения в `.csproj`;
- объяснить роль каждого пакета;
- не менять все endpoint-ы одновременно.

### Шаг 3. Создать контекст базы

- создать `WorkPulseDbContext`;
- добавить `DbSet<WorkTask>`;
- зарегистрировать контекст;
- добавить connection string.

Практическая причина для знакомства с dependency injection появится именно здесь.

### Шаг 4. Первая миграция

- создать миграцию;
- рассмотреть созданные файлы;
- применить миграцию;
- увидеть файл SQLite;
- посмотреть структуру таблицы.

### Шаг 5. Перенести чтение

Сначала заменить только `GET /tasks`, затем проверить пустую базу.

### Шаг 6. Перенести создание

- `POST /tasks`;
- ID назначает база;
- сохранить изменения;
- проверить `201 Created`;
- проверить сохранение после перезапуска.

### Шаг 7. Перенести изменение и удаление

- `PATCH /tasks/{id}/complete`;
- `DELETE /tasks/{id}`;
- асинхронные методы EF Core;
- `404` для отсутствующих записей.

### Шаг 8. Привести модель и API-контракт в порядок

После работающей базы:

- вынести `WorkTask` из `Program.cs`;
- обсудить сущность и DTO;
- запретить клиенту управлять ID;
- улучшить валидацию;
- убрать или исправить небезопасный `/tasks/first`.

## После базы

Ближайшие функциональные улучшения:

- описание задачи;
- дата создания;
- срок выполнения;
- приоритет;
- фильтрация выполненных задач;
- изменение названия;
- проекты и категории;
- пользователи.

Каждая возможность должна становиться поводом изучить конкретную часть C# или .NET, а не просто наращивать код.

## Правила темпа

- одна новая крупная идея за раз;
- не повторять уже понятую тему без причины;
- при пробеле в базе C# останавливаться и разбирать его на текущем коде;
- не завершать занятие самостоятельно — ученик сам сообщает, когда остановиться;
- сложные термины давать только вместе с практическим примером;
- после ошибки сначала исследовать её, а не сразу скрывать готовым исправлением.
