# План занятий WorkPulse

Обновлено: **2026-07-22**

## Формат работы

Работа строится короткими практическими итерациями:

1. Выбирается одна понятная задача.
2. Перед новым понятием даётся необходимое объяснение.
3. Ученик предлагает решение или прогнозирует результат.
4. Вносится минимальное изменение.
5. Выполняется подходящая проверка.
6. Ученик объясняет результат своими словами.
7. Наставник уточняет модель и исправляет неточности.
8. После законченного этапа предлагается название коммита.
9. Commit и push ученик выполняет самостоятельно.
10. После заметного этапа обновляется документация.

После того как закономерность понята, однотипные механические изменения можно выполнять сразу в нескольких местах и проверять общим прогоном.

Не задавать вопрос на знание конструкции, которая ещё не вводилась. Сначала кратко объяснить новое средство, затем дать применить его самостоятельно.

Если новая задача требует незнакомой инфраструктурной конструкции, явно отделять:

```text
что нужно понять сейчас
```

от:

```text
что пока используется как готовый шаблон
```

## Текущий этап: редактирование задачи

### Шаг 13. Выбор HTTP-семантики

Цель — осознанно выбрать способ изменения `Title` и `Description`.

Перед кодом разобрать:

```text
PUT
— полная замена представления ресурса

PATCH
— частичное изменение ресурса
```

Вопросы для решения:

1. Должен ли клиент передавать сразу `Title` и `Description`?
2. Можно ли менять только одно поле?
3. Нужно ли уметь явно очистить `Description`, установив `null`?
4. Чем отличается «поле не передали» от «передали null»?
5. Какой маршрут будет понятнее клиенту?

Результат шага:

- выбран `PUT` или `PATCH`;
- решение объяснено своими словами;
- зафиксирован ожидаемый JSON-контракт.

### Шаг 14. DTO запроса редактирования

После выбора семантики:

- создать отдельный DTO;
- не использовать `WorkTask` как входной контракт;
- определить поля и nullable-состояния;
- добавить валидацию заголовка;
- не менять `Id` и `IsCompleted` через этот DTO.

Возможное имя после выбора контракта:

```text
UpdateWorkTaskDto
```

Точное имя утвердить перед реализацией.

### Шаг 15. Endpoint редактирования

Реализовать:

- поиск задачи;
- `404 Not Found`, если задача отсутствует;
- проверку входных данных;
- изменение разрешённых свойств;
- `SaveChangesAsync`;
- ответ через `WorkTaskResponseDto`.

Не объединять этот endpoint с:

```text
PATCH /tasks/{id}/complete
```

без отдельного обсуждения ответственности маршрутов.

### Шаг 16. Интеграционный тест редактирования

Успешный сценарий:

```text
создать исходную задачу
→ отправить запрос редактирования
→ получить успешный статус
→ проверить DTO ответа
→ повторно прочитать строку из базы
→ убедиться, что поля изменились
```

Ошибочные сценарии добавлять постепенно:

- пустой новый `Title`;
- отсутствующий `Id`;
- особенности nullable-описания после выбора контракта.
