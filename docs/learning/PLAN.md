# План занятий WorkPulse

Обновлено: **2026-07-19**

## Формат каждой сессии

Работа строится короткими итерациями:

1. Выбирается одна небольшая продуктовая или техническая задача.
2. Вносится минимальное изменение или выполняется один небольшой эксперимент.
3. Приложение запускается.
4. Проверяется результат.
5. Ученик объясняет наблюдение своими словами.
6. Наставник уточняет модель происходящего и вводит только нужную теорию.
7. Выполняется сборка или другая подходящая проверка.
8. После законченного этапа предлагается название коммита.
9. Commit и push ученик выполняет самостоятельно.
10. После заметного этапа синхронизируются документы в `docs/learning`.

Не давать большой готовый блок кода, когда ученик может собрать изменение самостоятельно.

Не считать тему изученной только потому, что конструкция встретилась в коде.

Не повторять стандартные Git-команды без необходимости. Подробно разбирать Git только при ошибке, новом состоянии или новой теме.

## Завершённый этап: асинхронная работа с EF Core

CRUD переводился по одному маршруту:

1. `GET /tasks` — `ToListAsync`;
2. `GET /tasks/{id}` — `FindAsync`;
3. `POST /tasks` — `SaveChangesAsync`;
4. `PATCH /tasks/{id}/complete` — `FindAsync` и `SaveChangesAsync`;
5. `DELETE /tasks/{id}` — `FindAsync` и `SaveChangesAsync`;
6. `GET /tasks/first` — `FirstOrDefaultAsync`.

Разобрано:

- асинхронный обработчик Minimal API;
- `Task`, `async`, `await`;
- освобождение потока во время незавершённого I/O;
- продолжение на любом свободном потоке;
- отличие асинхронности от создания нового потока;
- польза асинхронности для Web API;
- последовательность зависимых операций;
- отсутствие необходимости использовать `await` для `Add`, `Remove` и изменения свойства.

Тема закреплена вопросами ученика. Повторную большую лекцию не начинать.

## Завершённый этап: SQL за операциями EF Core

### Шаг 6. Увидеть SQL за LINQ и CRUD

Выполнено:

- найден SQL EF Core в окне `Run`;
- разобрана цепочка `LINQ → EF Core → SQL → SQLite → C#-объекты`;
- `GET /tasks` сопоставлен с `SELECT`;
- разобраны `SELECT`, список столбцов, `FROM` и псевдоним таблицы;
- добавлен временный `OrderBy`, в SQL найден `ORDER BY`;
- выполнен эксперимент с отложенным выполнением;
- подтверждено, что `OrderBy` и `Where` строят запрос;
- подтверждено, что `ToListAsync` выполняет запрос;
- временный `Where` сопоставлен с `WHERE`;
- `FirstOrDefaultAsync` сопоставлен с `LIMIT 1`;
- отсутствие результата вернуло `null`;
- `POST` сопоставлен с `INSERT`;
- разобраны SQL-параметры `@p0`, `@p1`;
- объяснено, что коллекция параметров хранится во временном объекте команды в памяти;
- `RETURNING "Id"` сопоставлен с получением созданного идентификатора;
- `PATCH` сопоставлен с `SELECT`, затем `UPDATE`;
- подтверждено, что EF Core обновляет только изменённое свойство;
- `DELETE` сопоставлен с `SELECT`, затем `DELETE`;
- разобрана критическая роль `WHERE` для `UPDATE` и `DELETE`;
- все временные изменения `GET /tasks` удалены;
- постоянного изменения production-кода после экспериментов не осталось.

### Учебный результат шага

Ученик может своими словами объяснить:

```text
db.Tasks               — источник данных
Where / OrderBy         — построение запроса
ToListAsync             — выполнение и список
FirstOrDefaultAsync     — выполнение и первый объект/null
Add                     — состояние Added
изменение свойства      — состояние Modified
Remove                  — состояние Deleted
SaveChangesAsync        — INSERT / UPDATE / DELETE
```

## Текущий этап: ручная практика SQL

### Шаг 7. Выполнить базовые SQL-команды без EF Core

Цель — подтвердить понимание уже увиденного SQL. Не превращать шаг в большой курс.

Порядок:

1. Выбрать удобный способ выполнить SQL для `workpulse.db`.
2. Выполнить:

```sql

FROM "Tasks";
```

3. Добавить фильтрацию через `WHERE`.
4. Добавить сортировку через `ORDER BY`.
5. Создать одну тестовую задачу через `INSERT`.
6. Изменить только её через `UPDATE ... WHERE`.
7. Удалить только её через `DELETE ... WHERE`.
8. Сравнить ручные команды с логами EF Core.
9. Ученик объясняет роль каждой части SQL.

Ограничения:

- не выполнять `UPDATE` или `DELETE` без `WHERE`;
- использовать только тестовую запись;
- не менять production-код ради ручного SQL;
- двигаться по одной команде.
