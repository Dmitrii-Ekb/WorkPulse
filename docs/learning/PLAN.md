# План занятий WorkPulse

## Формат каждой сессии

Работа строится короткими итерациями:

1. Выбирается одна небольшая продуктовая или техническая задача.
2. Вносится минимальное изменение в проект.
3. Приложение запускается.
4. Проверяется успешный сценарий и хотя бы один ошибочный.
5. Ученик объясняет наблюдение своими словами.
6. Наставник уточняет модель происходящего и вводит только нужную теорию.
7. Результат фиксируется коммитом после законченного шага.

Не давать сразу большой готовый блок кода, когда ученик уже знает необходимые части и может собрать решение самостоятельно.

Не считать конструкцию изученной только потому, что она встретилась в готовом коде.

## Ближайший план

### Шаг 1. Перенести строку подключения в конфигурацию

Практическая задача:

- добавить секцию `ConnectionStrings` в `appsettings.json`;
- дать строке подключения понятное имя;
- прочитать её через `builder.Configuration.GetConnectionString(...)`;
- передать полученное значение в `UseSqlite`;
- проверить запуск и CRUD.

Теория только после практического изменения:

- зачем отделять настройки от кода;
- источники конфигурации ASP.NET Core;
- ключи и секции JSON;
- что возвращает `GetConnectionString`;
- почему секреты и окружения потребуют другого подхода позже.

### Шаг 2. Провести маленький эксперимент с ошибкой конфигурации

- временно изменить имя строки подключения;
- запустить приложение;
- прочитать сообщение об ошибке;
- вернуть правильное значение;
- своими словами объяснить связь между ключом конфигурации и `UseSqlite`.

### Шаг 3. Вынести модель из `Program.cs`

- создать отдельный файл для `WorkTask`;
- сохранить текущее пространство имён;
- убедиться, что `Program.cs` стал короче;
- проверить, что миграция и CRUD продолжают работать без изменения схемы.

Цель шага — познакомиться с организацией кода, не вводя преждевременно слои и сложную архитектуру.

### Шаг 4. Отделить входной контракт создания от сущности базы

Практическая причина: сейчас `POST /tasks` принимает `WorkTask`, поэтому клиент видит поля, которыми не должен управлять напрямую.

- создать простой тип запроса на создание задачи;
- принимать только нужные входные данные;
- самостоятельно создавать `WorkTask`;
- проверить, что клиент не задаёт `Id`;
- обсудить DTO только на этой конкретной проблеме.

### Шаг 5. Перевести операции EF Core на асинхронные методы

Переносить по одному маршруту:

- `ToListAsync`;
- `FindAsync`;
- `SaveChangesAsync`;
- `FirstOrDefaultAsync`.

На каждом шаге:

- добавить `async` и `await`;
- запустить приложение;
- проверить поведение;
- объяснить, почему Web API обычно не должен блокировать поток во время ожидания базы.

Не превращать этот шаг в большой абстрактный курс по многопоточности.

### Шаг 6. Увидеть SQL за LINQ

- включить или прочитать существующие логи EF Core;
- сопоставить `ToList` с `SELECT`;
- `Add` и `SaveChanges` — с `INSERT`;
- изменение и `SaveChanges` — с `UPDATE`;
- `Remove` и `SaveChanges` — с `DELETE`;
- написать несколько простых SQL-команд вручную отдельно от production-кода.

### Шаг 7. Вторая миграция на реальном изменении модели

Выбрать одно небольшое полезное поле, например:

- описание задачи;
- дата создания;
- срок выполнения;
- приоритет.

Затем:

1. изменить модель;
2. создать новую миграцию;
3. прочитать `Up` и `Down`;
4. применить миграцию;
5. проверить старые и новые данные;
6. объяснить переход схемы из первой версии во вторую.

## После ближайшего этапа

Функциональные улучшения:

Каждая возможность должна становиться поводом изучить конкретную часть C# или .NET, а не просто наращивать код.

## Правила темпа

- одна новая крупная идея за раз;
- не повторять уже понятую тему без причины;
- при пробеле в базе C# останавливаться и разбирать его на текущем коде;
- новые конструкторы, обобщения, наследование и DI считать только введёнными, пока ученик не сможет объяснить и изменить их;
- не завершать занятие самостоятельно — ученик сам сообщает, когда остановиться;
- сложные термины давать только вместе с практическим примером;
- после ошибки сначала исследовать её, а не сразу скрывать готовым исправлением;
- не вводить репозитории, сервисные слои или контроллеры без конкретной проблемы, которую они решают.
