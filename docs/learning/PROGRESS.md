# Прогресс обучения WorkPulse

Обновлено: **2026-07-19**

## Текущая точка

Minimal API для задач работает с SQLite через Entity Framework Core. Данные сохраняются после перезапуска приложения.

Строка подключения хранится в `appsettings.json`, читается через `GetConnectionString("WorkPulseDatabase")` и проверяется при запуске приложения.

Модель базы данных `WorkTask` вынесена из `Program.cs` в отдельный файл `WorkTask.cs`.

Для `POST /tasks` используется входной DTO `CreateWorkTaskDto`. Клиент передаёт только `Title`, а сервер самостоятельно создаёт сущность `WorkTask`, устанавливает `IsCompleted = false` и позволяет базе назначить `Id`.

Весь текущий CRUD переведён на асинхронные операции EF Core:

- `GET /tasks` использует `ToListAsync`;
- `GET /tasks/first` использует `FirstOrDefaultAsync`;
- `GET /tasks/{id}` использует `FindAsync`;
- `POST /tasks` использует `SaveChangesAsync`;
- `PATCH /tasks/{id}/complete` использует `FindAsync` и `SaveChangesAsync`;
- `DELETE /tasks/{id}` использует `FindAsync` и `SaveChangesAsync`.

Проверено, что HTTP-поведение маршрутов не изменилось:

- получение существующих данных возвращает `200 OK`;
- отсутствующая задача возвращает `404 Not Found`;
- создание возвращает `201 Created`;
- некорректное название по-прежнему возвращает `400 Bad Request`;
- удаление возвращает `204 No Content`;
- повторное получение или удаление удалённой задачи возвращает `404 Not Found`.

Текущая учебная точка — **закрепление асинхронности перед переходом к новой теме**.

Нужно спокойно разобрать вопросы про `Task`, `async`, `await`, поток, пул потоков, ожидание ввода-вывода и продолжение метода после завершения операции.

Следующая запланированная тема — посмотреть, какой SQL формирует EF Core. Она не начинается, пока вопросы по текущему этапу не разобраны.

## Текущее состояние проекта

### Модель базы данных

Сущность находится в отдельном файле `WorkTask.cs`:

```csharp
namespace WorkPulse;

public class WorkTask
{
    public int Id { get; set; }
    public string Title { get; set; } = "";
    public bool IsCompleted { get; set; }
}
```

`WorkTask` описывает данные задачи, которые сохраняются в таблице `Tasks`.

Класс остаётся `public`, потому что он используется публичным свойством `DbSet<WorkTask>` в контексте базы.

### Входной DTO

В отдельном файле `CreateWorkTaskDto.cs` находится контракт запроса на создание задачи:

```csharp
namespace WorkPulse;

public class CreateWorkTaskDto
{
    public string Title { get; set; } = "";
}
```

`CreateWorkTaskDto` не является сущностью базы данных и не входит в `DbSet`.

Его задача — описать только те данные, которые разрешено передавать клиенту при создании задачи.

Текущая цепочка создания:

```text
JSON
  ↓
CreateWorkTaskDto request
  ↓ валидация и явное преобразование
WorkTask newTask
  ↓
EF Core
  ↓
SQLite
```

Подключены пакеты:

```csharp
public class WorkPulseDbContext : DbContext
```


- при неправильном или отсутствующем ключе приложение останавливается при запуске с понятной ошибкой.

### Миграции

Создана первая миграция:

```text

```

В методе `Up` она создаёт таблицу `Tasks`:

- `Id` — `INTEGER`, первичный ключ, `AUTOINCREMENT`;
- `Title` — `TEXT NOT NULL`;
- `IsCompleted` — `INTEGER NOT NULL`.

В методе `Down` таблица `Tasks` удаляется.

Миграция применена командой:

```text

```

- служебная таблица блокировки миграций.

### Работа маршрутов с базой

`GET /tasks`:

```csharp
app.MapGet("/tasks", async (WorkPulseDbContext db) =>
{
    var tasks = await db.Tasks.ToListAsync();

    return tasks;
});
```

`ToListAsync` асинхронно получает все задачи из таблицы.

`GET /tasks/first`:

```csharp
app.MapGet("/tasks/first", async (WorkPulseDbContext db) =>
{
    var task = await db.Tasks
        .OrderBy(task => task.Id)
        .FirstOrDefaultAsync();

    if (task is null)
    {
        return Results.NotFound();
    }

    return Results.Ok(task);
});
```

`OrderBy` только формирует описание запроса. Обращение к SQLite выполняется при вызове `FirstOrDefaultAsync`.

`GET /tasks/{id}`:

```csharp
app.MapGet("/tasks/{id}", async (
    int id,
    WorkPulseDbContext db) =>
{
    var task = await db.Tasks.FindAsync(id);

    if (task is null)
    {
        return Results.NotFound();
    }

    return Results.Ok(task);
});
```

`FindAsync` ищет задачу по первичному ключу и возвращает найденный `WorkTask` или `null`.

`POST /tasks`:

```csharp
app.MapPost("/tasks", async (
    CreateWorkTaskDto request,
    WorkPulseDbContext db) =>
{
    if (string.IsNullOrWhiteSpace(request.Title))
    {
        return Results.BadRequest(
            "Название задачи обязательно");
    }

    var newTask = new WorkTask
    {
        Title = request.Title,
        IsCompleted = false
    };

    db.Tasks.Add(newTask);

    await db.SaveChangesAsync();

    return Results.Created(
        $"/tasks/{newTask.Id}",
        newTask);
});
```

`Add` только добавляет новую сущность в отслеживание EF Core. Реальная команда `INSERT` отправляется в SQLite при `SaveChangesAsync`.

`PATCH /tasks/{id}/complete`:

```csharp
app.MapPatch("/tasks/{id}/complete", async (
    int id,
    WorkPulseDbContext db) =>
{
    var task = await db.Tasks.FindAsync(id);

    if (task is null)
    {
        return Results.NotFound();
    }

    task.IsCompleted = true;

    await db.SaveChangesAsync();

    return Results.Ok(task);
});
```

Поиск и сохранение выполняются последовательно:

1. `FindAsync` получает сущность;
2. свойство меняется в C#-объекте;
3. `SaveChangesAsync` отправляет `UPDATE` в SQLite.

`DELETE /tasks/{id}`:

```csharp
app.MapDelete("/tasks/{id}", async (
    int id,
    WorkPulseDbContext db) =>
{
    var task = await db.Tasks.FindAsync(id);

    if (task is null)
    {
        return Results.NotFound();
    }

    db.Tasks.Remove(task);

    await db.SaveChangesAsync();

    return Results.NoContent();
});
```

`Remove` только помечает сущность состоянием `Deleted`. Реальная команда `DELETE` выполняется при `SaveChangesAsync`.

Временный маршрут `/db-test` удалён после проверки провайдера.

Старый `List<WorkTask>` полностью удалён из `Program.cs`.

## Проверенное поведение

- пустое название отклоняется с `400 Bad Request`;
- название из пробелов отклоняется с `400 Bad Request`;
- отсутствующая задача возвращает `404 Not Found`;
- создание возвращает `201 Created`;
- удаление возвращает `204 No Content`;
- `Id` назначается SQLite автоматически;
- `IsCompleted` при создании устанавливается сервером в `false`;
- клиент не может задать `Id` или начальное значение `IsCompleted` через DTO;
- созданная задача доступна через `GET`;
- изменение `IsCompleted` сохраняется в базе;
- удалённая задача исчезает из базы;
- данные остаются после перезапуска приложения;
- пустой `/tasks/first` возвращает `404`, а не вызывает исключение;
- неверный ключ строки подключения останавливает приложение с понятной ошибкой;
- `ToListAsync`, `FindAsync`, `FirstOrDefaultAsync` и `SaveChangesAsync` сохраняют прежний HTTP-контракт;
- асинхронные маршруты корректно обрабатывают успешные и ошибочные сценарии;
- проект собирается после перевода CRUD на асинхронные операции.

## Проведённые эксперименты и найденные ошибки

1. Использование `tasks[id - 1]` перепутало ID сущности с индексом массива.
2. Запрос несуществующего индекса вызвал `IndexOutOfRangeException`.
3. Изменение ID на `20` показало, почему поиск должен идти по свойству `Id`.
4. `FirstOrDefault` вернул объект или `null`.
5. Попытка изменить свойство анонимного объекта показала его неизменяемость.
6. Массив не позволил вызвать `Add`, после чего использован `List<WorkTask>`.
7. POST без пустой строки после заголовка был ошибочно разобран HTTP-клиентом.
8. Поле `name` вместо `title` не заполнило `Title`.
9. Валидация после добавления была бы слишком поздней.
10. Клиентский ID позволял создавать некорректные значения при хранении в списке.
11. `FirstOrDefault` при дубликатах вернул первую найденную задачу.
12. `Max` на пустом списке не смог вычислить значение.
13. Изменение объекта через локальную переменную изменило тот же объект в списке.
14. После перезапуска сервера данные из списка исчезли.
15. Пакет `Microsoft.EntityFrameworkCore.Design` сам по себе не устанавливает команду `dotnet ef`.
16. Для CLI отдельно установлен `dotnet-ef` той же версии `9.0.17`.
17. Первая миграция показала сгенерированные `CreateTable`, `PrimaryKey`, `Up` и `Down`.
18. После `database update` SQLite создал таблицу точно по миграции.
19. После перехода на SQLite данные сохранились после перезапуска.
20. Без `SaveChanges` подготовленное добавление, изменение или удаление не записывается в базу.
21. Старая реализация `/tasks/first` могла падать; новая возвращает безопасный `404`.
22. Неверное имя строки подключения привело к ранней и понятной ошибке запуска.
23. Перенос `WorkTask` в отдельный файл не изменил модель EF Core и не потребовал миграции.
24. После замены параметра POST с `WorkTask` на `CreateWorkTaskDto` компилятор не разрешил передать DTO в `DbSet<WorkTask>`.
25. Явное создание `new WorkTask` решило несовпадение типов и отделило входной контракт от сущности.
26. Отправка `id` и `isCompleted` в JSON показала, что DTO принимает только объявленное свойство `Title`.
27. POST с пустым объектом `{}` создал DTO, у которого `Title` остался пустой строкой, после чего сработала валидация.
28. Добавление DTO не потребовало миграции, потому что таблица `Tasks` не изменилась.
29. `GET /tasks` после перехода на `ToListAsync` вернул тот же список и тот же HTTP-ответ.
30. `FindAsync` корректно вернул существующую задачу и `null` для отсутствующей.
31. `POST /tasks` с `SaveChangesAsync` сохранил задачу и получил назначенный SQLite идентификатор.
32. Валидация POST завершает обработчик раньше, поэтому `SaveChangesAsync` не вызывается для некорректного названия.
33. В PATCH сначала выполняется `FindAsync`, затем изменение объекта, затем `SaveChangesAsync`; операции зависят друг от друга и идут последовательно.
34. `Remove` не удаляет строку немедленно: удаление выполняется только после `SaveChangesAsync`.
35. Первый DELETE вернул `204`, а последующие GET и DELETE той же задачи вернули `404`.
36. `FirstOrDefaultAsync` сохранил безопасное поведение маршрута `/tasks/first`.
37. Разобрано, что продолжение после `await` не обязано выполняться тем же потоком, который начал обработчик.

- механизм продолжения метода после завершения ожидаемой операции;
- отсутствие гарантии возврата на тот же поток после `await`;
- отличие асинхронности от параллелизма и явного создания нового потока.

## Что пока не считать изученным

- написание SQL вручную;
- внешние ключи на практике;
- связи таблиц;
- `JOIN`;
- индексы;
- транзакции;
- уникальные ограничения;
- окружения ASP.NET Core;
- значимые типы в сравнении со ссылочными;
- методы класса как отдельная тема;
- инкапсуляция;
- полиморфизм;
- интерфейсы;
- middleware;
- контроллеры;
- автоматическое тестирование;
- авторизация;
- Docker и развёртывание;
- архитектурные паттерны.

## Уровень понимания и стиль обучения

Исходный уровень:

- основы синтаксиса C# знакомы;
- ООП почти с нуля;
- Web API и ASP.NET Core с нуля;
- Git на начальном уровне;
- архитектура почти с нуля.

Предпочтительный подход:

- практический проект вместо абстрактного курса;
- сначала изменение и эксперимент, потом объяснение;
- небольшие шаги;
- объяснение простыми словами и аналогиями;
- ученик регулярно формулирует происходящее своими словами;
- не считать тему освоенной после простого копирования кода;
- при новом синтаксисе останавливаться и разбирать только необходимую часть;
- не перескакивать к сложной архитектуре раньше времени;
- не заканчивать сессию без решения ученика;
- после завершённого этапа наставник предлагает имя коммита, а ученик самостоятельно выполняет `commit` и `push`;
- подробно останавливаться на Git-командах только при новой теме Git или при возникновении ошибки.

## Следующий конкретный шаг

Не вносить новые изменения в код до завершения вопросов по асинхронности.

Закрепить на текущих маршрутах:

1. что такое поток выполнения;
2. что такое пул потоков;
3. какой поток начинает обрабатывать HTTP-запрос;
4. что возвращают `ToListAsync`, `FindAsync` и `SaveChangesAsync`;
5. что именно делает `await`, если операция ещё не завершена;
6. кто сообщает `Task`, что операция ввода-вывода завершилась;
7. как сохраняется точка продолжения метода;
8. почему продолжение может выполнить другой свободный поток;
9. почему `Add`, `Remove` и изменение свойства не требуют `await`;
10. когда асинхронность полезна Web API и почему в маленьком учебном приложении разница почти незаметна.

После ответов на вопросы:

- выполнить `dotnet build`, если он ещё не выполнен после последних изменений;
- зафиксировать асинхронный CRUD отдельным коммитом;
- только затем перейти к следующей теме — SQL, который формирует EF Core.
