# Карта обучения WorkPulse

Обновлено: **2026-07-19**

## Цель маршрута

Через последовательное развитие WorkPulse научиться самостоятельно проектировать, писать, отлаживать, тестировать и развёртывать .NET-сервисы.

Конечная цель — уверенный уровень Middle C#/.NET-разработчика и проект, который можно содержательно обсуждать на собеседовании.

Маршрут адаптивный и не привязан к жёстким срокам. Следующая тема начинается после практического понимания предыдущей.

## Текущая веха

WorkPulse имеет работающий асинхронный Minimal API и постоянное хранение задач в SQLite через EF Core.

Работает вертикальная цепочка:

```text
HTTP-запрос
    ↓
DTO и валидация
    ↓
асинхронный endpoint
    ↓
сущность WorkTask
    ↓
DbContext и Change Tracker
    ↓
LINQ или изменение состояния сущности
    ↓
SQL-команда с параметрами
    ↓
SQLite
    ↓
C#-объекты и HTTP-ответ
```

Завершены ближайшие улучшения:

- строка подключения вынесена в конфигурацию;
- обязательная настройка проверяется при запуске;
- `WorkTask` вынесен из `Program.cs`;
- создан `CreateWorkTaskDto`;
- входной контракт отделён от сущности;
- клиент не управляет `Id` и начальным `IsCompleted`;
- весь CRUD переведён на асинхронные методы;
- вопросы по базовой механике `async`/`await` разобраны;
- найден и прочитан SQL EF Core;
- доказано отложенное выполнение LINQ;
- `SELECT`, `INSERT`, `UPDATE`, `DELETE` сопоставлены с кодом приложения;
- разобраны параметры SQL и получение созданного `Id`.

Текущая контрольная точка:

```text
асинхронный CRUD работает
        ↓
SQL EF Core прочитан и объяснён
        ↓
написать базовые SQL-команды вручную
        ↓
создать вторую миграцию
```

## Ближайшая последовательность

```text
1. Выполнить базовые SQL-команды вручную
2. Привести WorkPulse.http к повторяемому виду
3. Добавить полезное поле в WorkTask
4. Создать и применить вторую миграцию
5. Решить вопрос входного и выходного DTO
```

Порядок может корректироваться, но без перескакивания через непонятые основы.
