# Дорожная карта обучения C# и .NET через WorkPulse

Обновлено: **2026-07-22**

## Цель

За шесть месяцев перейти от начального уровня к уверенной подготовке на позицию Middle C#/.NET-разработчика.

Обучение строится не как набор изолированных лекций, а через развитие реального продукта WorkPulse.

Целевой результат:

- уверенно писать и читать C#;
- понимать ASP.NET Core и HTTP;
- проектировать Web API;
- работать с EF Core и SQL;
- писать автоматические тесты;
- понимать ООП и основные архитектурные принципы;
- работать с Git;
- диагностировать ошибки;
- объяснять решения на собеседовании;
- иметь проект, который показывает практические навыки.

## Принципы

- 2–3 часа в день в среднем;
- одна основная практическая задача за итерацию;
- теория вводится перед реальной необходимостью;
- пятница — рефакторинг и закрепление;
- суббота — мини-собеседование;
- регулярный code review;
- документация хранит актуальный контекст;
- Git-коммиты фиксируют законченные этапы;
- сложность растёт постепенно;
- преждевременная архитектура не добавляется.

## Текущая позиция

Пройден ранний backend-фундамент:

```text
Minimal API
→ HTTP и CRUD
→ EF Core и SQLite
→ миграции
→ DTO
→ async/await
→ LINQ и SQL
→ первые интеграционные тесты
```

Текущая точка:

```text
интеграционные тесты создания задачи завершены
→ следующий этап: редактирование Title и Description
```

## Этап 1. База C# внутри простого API

Статус: **завершён на текущем уровне**

- коды HTTP;
- `null`;
- базовая валидация.

Статус: **завершён на базовом уровне**

## Этап 5. Автоматическое тестирование API

Статус: **первый подэтап завершён**

Темы:

- автоматический тест;
- unit- и integration-тест;
- Arrange, Act, Assert;
- xUnit;
- `WebApplicationFactory`;
- тестовый `HttpClient`;
- тестовая SQLite in-memory;
- изоляция данных;
- успешный и ошибочный сценарий;
- проверка ответа и базы.

Практический результат:

- `GET /hello` проверяется автоматически;
- корректный `POST /tasks` проверяется автоматически;
- пустой `Title` проверяется автоматически;
- тесты повторно проходят независимо.

Дальнейшее развитие тестов выполняется вместе с новыми функциями, а не отдельным массовым покрытием всего CRUD.

## Этап 6. Редактирование ресурса и HTTP-семантика

Статус: **следующий**

Темы:

- `PUT` против `PATCH`;
- полная и частичная замена;
- DTO редактирования;
- nullable-поля в контракте;
- `404`;
- валидация;
- интеграционный тест обновления.

Практический результат:

- изменение `Title`;
- изменение `Description`;
- стабильный ответ через `WorkTaskResponseDto`;
- тест, доказывающий изменение в базе.

## Этап 7. Системное ООП

Статус: **предстоит**

Отдельно пройти:

- класс и объект;
- поле и свойство;
- конструктор;
- модификаторы доступа;
- инкапсуляция;
- ответственность класса;
- композиция;
- наследование;
- полиморфизм;
- интерфейсы;
- абстрактные классы;
- `static`;
- `readonly`;
- `record` и обычный класс.

Закреплять на рефакторинге WorkPulse, не создавая искусственную архитектуру.

## Этап 8. Прикладная архитектура

Статус: **предстоит**

- extension-методы;
- сервисный слой;
- repository — только после обсуждения пользы и цены;
- DI;
- времена жизни сервисов;
- options pattern;
- обработка ошибок;
- логирование.

## Этап 9. Расширение модели

Статус: **предстоит**

Добавлять по одному продуктово полезному полю:

- `CreatedAt`;

- `Priority`;

```text

→ C#-тип
→ сущность
→ миграция
→ DTO
→ API
→ тест
```

## Ориентировочная последовательность ближайших блоков

```text
1. Редактирование задачи
2. Интеграционный тест редактирования
3. Системный блок ООП на коде WorkPulse
4. Следующее поле модели и миграция
5. Фильтрация и сортировка
6. Пагинация
7. Рефакторинг endpoint-ов

```

Последовательность может меняться по результатам практики и мини-собеседований, но темы не должны пропадать.
