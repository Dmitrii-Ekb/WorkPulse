# Карта обучения WorkPulse

## Цель маршрута

Не просто собрать учебный API, а научиться самостоятельно проектировать, писать, отлаживать и развивать .NET-сервисы. Конечная цель — уверенная база уровня middle-разработчика и проект, который не стыдно обсуждать на собеседовании.

Маршрут не привязан к жёстким срокам. Переход к следующей теме происходит после практического понимания предыдущей.

## Текущая веха

WorkPulse уже имеет Minimal API и постоянное хранение задач в SQLite через EF Core.

Завершена первая вертикальная цепочка:

```text
HTTP-запрос → endpoint → DbContext → EF Core → SQLite → HTTP-ответ
```

Следующая задача — вынести строку подключения в конфигурацию, затем привести модель и входной API-контракт в порядок.

## Этап 1. База C# через реальный код

Статус: **в процессе**

Уже применялось:

- переменные, типы и выражения;
- условия;
- массивы и `List<T>`;
- классы, объекты и свойства;
- `null`;
- лямбды и базовый LINQ;
- строки;
- обобщения встречены в коде EF Core.

Ещё предстоит закрепить:

- методы;
- циклы;
- конструкторы;
- ссылочные и значимые типы;
- nullable-типы;
- исключения;
- делегаты;
- даты и перечисления;
- обобщения;
- `async`/`await`;
- управление памятью на практическом уровне.

Результат: ученик уверенно читает и пишет обычный C#-код и может объяснить, что происходит с данными и объектами.

## Этап 3. Базы данных и SQL

Уже применялось:

- постоянное хранение данных;

Промежуточный результат уже достигнут: данные WorkPulse сохраняются после перезапуска.

Полный результат этапа: данные корректно связаны между собой, а ученик умеет читать и писать основные SQL-запросы.

## Этап 4. Entity Framework Core

Статус: **в процессе**

Уже применялось:

- `DbSet`;
- строка подключения;
- миграции;
- `Up` и `Down`;
- история миграций;
- базовое отслеживание изменений;

- `Find`;
- полный синхронный CRUD.

Ближайшие темы:

- конфигурация строки подключения;
- асинхронные операции;
- более явное понимание сгенерированного SQL;
- вторая миграция;

- границы транзакций;

Промежуточный результат уже достигнут: API работает с реальной базой через EF Core.

Полный результат этапа: ученик понимает не только вызовы EF Core, но и какой SQL и какие изменения базы примерно стоят за ними.

## Этап 5. ООП и организация кода

Статус: **следующий после ближайшей работы с конфигурацией**

Темы вводятся после появления конкретной неудобной части кода:

- вынесение `WorkTask` из `Program.cs`;
- методы и ответственность;
- инкапсуляция;
- DTO и отделение API-контракта от сущности базы;
- интерфейсы;
- композиция и наследование;
- endpoint-модули или контроллеры при необходимости;
- сервисы;
- dependency injection глубже текущего минимального использования;
- репозитории только при реальной необходимости;
- базовые принципы SOLID на конкретных проблемах проекта.

Результат: код можно развивать без превращения проекта в один большой файл.
